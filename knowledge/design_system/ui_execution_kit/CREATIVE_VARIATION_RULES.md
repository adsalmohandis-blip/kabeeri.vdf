# Creative Variation Rules

Kabeeri must not output the same design for every similar app. Consistency controls quality; variation controls product fit.

## Fixed Quality Rules

These never change:

- approved UI foundation;
- semantic tokens;
- accessible buttons, labels, focus states, and keyboard behavior;
- loading, empty, error, success, disabled states;
- responsive behavior;
- RTL readiness when Arabic is supported;
- no raw colors inside page components;
- no custom duplication of approved components.

## Variable Creative Axes

For every project, vary at least four axes based on questionnaire answers:

- palette preset and accent strategy;
- typography mood: compact, editorial, friendly, formal, technical;
- density: comfortable, balanced, compact;
- page rhythm: list-first, card-first, command-first, chart-first, content-first;
- navigation pattern: top nav, sidebar, split nav, tabs, mobile bottom nav;
- surface style: flat, bordered, soft shadow, strong section bands, dark shell;
- component emphasis: tables, cards, forms, timelines, kanban, charts, media;
- empty state tone: operational, encouraging, formal, premium, playful;
- dashboard hierarchy: KPI-first, task-first, alert-first, workflow-first;
- microcopy tone: concise, warm, formal, technical, bilingual.

## Variation Inputs

Use these inputs before generating UI:

- business domain;
- target user role;
- brand personality;
- Arabic/English requirement;
- data density;
- primary workflow;
- risk/compliance level;
- public SEO or private dashboard;
- selected UI foundation;
- approved reference pattern.

## Anti-Repetition Rule

If two developers ask for the same app type, Kabeeri can share the same acceptance rules but should vary:

- palette preset;
- layout hierarchy;
- component emphasis;
- density level;
- microcopy tone;
- dashboard information order;
- navigation style when product scope allows it.

## Creativity Boundary

Creativity is allowed in:

- composition;
- spacing rhythm;
- visual hierarchy;
- illustration/media choice;
- tone;
- dashboard ordering;
- alternate states and empty-state messaging.

Creativity is not allowed in:

- inventing unapproved libraries;
- ignoring tokens;
- removing required states;
- replacing accessible controls with decorative elements;
- using raw colors in components;
- changing business logic during UI refactors.
