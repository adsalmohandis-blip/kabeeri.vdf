# Confirmation And Status Copy

## Confirmations

Destructive confirmations must include:

- the object affected;
- the consequence;
- whether the action can be undone;
- the safer alternative when available.

## Status Labels

Use stable status vocabulary across the product.

Examples:

- Draft
- Pending
- Active
- Paused
- Approved
- Rejected
- Failed
- Processing
- Completed
- Archived

## Feedback

- Success messages should state what changed.
- Warning messages should explain risk.
- Info messages should explain context.
- Error messages should explain recovery.
- Toasts are for temporary feedback; alerts are for persistent or blocking states.
