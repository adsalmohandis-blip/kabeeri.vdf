# Component Composition Contract

A composition is a reusable screen-level arrangement of components, states, and
review expectations. It is not a visual clone. It is a governed starting point.

## Every Composition Defines

- purpose
- best product types
- primary components
- supporting components
- required states
- responsive behavior
- accessibility focus
- performance focus
- microcopy focus
- visual QA focus

## Rules

- Pick one primary composition per screen before implementation.
- Do not stack unrelated compositions on one page.
- Adapt density, palette, copy, and component variants from product answers.
- Use the selected UI foundation's components first.
- Create component contracts for repeated or business-critical components.
- Include empty, loading, error, success, disabled, and validation states when relevant.
- Reference the composition ID in prompts instead of restating the full structure.
