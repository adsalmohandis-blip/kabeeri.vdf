Framework Development

Framework development means changing KVDF itself safely: runtime, commands, dashboards, schemas, docs, reports, and tests move together.

Beginner note

Start here when you want the shortest route into Framework Development.

Topic Details

Change runtime

Update commands, services, state files, and runtime behavior together.

Keep docs aligned

Whenever capability changes, update the canonical docs and the site pages with it.

Update tests

A framework change is not complete until the coverage proves the behavior.

Protect boundaries

Framework work should not leak into app workspace behavior or app docs.

Release discipline

Framework changes should be reviewed, verified, and only then considered ready.

What to touch

Commands, dashboard, validation, schemas, reports, and plugins are the main framework surfaces.

How to think about it

A framework change usually means changing runtime behavior, public docs, and evidence outputs together.

What to avoid

Do not fix only the page, only the command, or only the state file. The framework must stay coherent across layers.

Practical examples

Adding a command, changing a dashboard route, updating the validation schema, or modifying the task pipeline are all framework changes.
