Kabeeri in 4 Pillars

Kabeeri can be understood through four pillars: the flow chains of the system, the main modules, the modules relationship, and the patterns of relationship. Together they show how the framework moves, how its capabilities are built, how those capabilities cooperate, and how recurring relationship shapes keep the system coherent.

Topic Details

The flow chains of the system

This pillar is the cleanest way to think about Kabeeri. Follow the request as it enters the system, see how the framework decides, watch how state changes, and track how outputs and evidence are produced. The goal is not to list every file; it is to show the end-to-end path the framework takes when it works correctly. If you can describe the flow chain, you can explain why the framework behaves the way it does, where the next decision happens, and what must stay in sync for the chain to remain trustworthy.

Main modules

These are the framework capabilities that make the system useful in practice: commands, services, validators, dashboards, generators, site builders, and report emitters. A main module is not just code; it is a capability the framework exposes or depends on. This pillar helps you understand which capabilities exist, which ones are core, and how the major building blocks turn policy into visible runtime behavior.

Modules relationship

This pillar explains how modules work together. A command may call a service, a service may depend on a validator, a dashboard may read from workspace state, and a docs page may mirror the command registry. The important idea is cooperation: modules do not live alone, and a change in one place can affect several others. When you understand these relationships, you can predict the effect of a change before it creates drift.

Patterns of relationship

This pillar captures the repeatable shapes that show up across the framework again and again. The system keeps repeating patterns such as source to validator, validator to report, state to dashboard, command to docs, and policy to execution. Recognizing the pattern is more useful than noticing a single dependency because it tells you how the framework behaves as a design system. Once you can name the pattern, you can decide whether a new change belongs to an existing shape or needs a new one.
