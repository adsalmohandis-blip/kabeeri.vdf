Modules Relationship

Module relationships explain how KVDF parts cooperate. A command can call a service, a service can depend on a validator, a dashboard can read workspace state, and a docs page can mirror the command registry.

Topic Details

Module relationship flow

Step 1: A session starts with resume, init, or entry.

Step 2: The session is routed into the right track and workspace state.

Step 3: The AI gathers the goal, then the profile router and delivery advisor classify the project.

Step 4: Blueprints, questionnaires, prompt packs, and design/data modules turn the idea into concrete planning assets.

Step 5: Tasks are generated, assessed, covered, and approved.

Step 6: Tokens, locks, and execution scope make sure only the right task can run.

Step 7: The task packet and executor contract narrow the AI boundary.

Step 8: Batch execution runs the approved work.

Step 9: Dashboard, reports, and audit trails keep the state visible.

Step 10: Verification, completion, handoff, and release close the loop.

Step 11: Plugins, multi-AI governance, and Evolution Steward extend the system itself when Kabeeri needs to evolve.

When you read a module relationship, think in terms of a chain that moves from session entry to routing, classification, planning assets, approved tasks, controlled execution, visible state, and finally extension of the system itself. The relationship is healthy when each module still fits the next one after the work runs, not only when the idea looks good on paper.

Command to service

Commands are usually the entry points, and services are the reusable workers behind them. This relationship keeps the CLI thin while the actual behavior stays in a place that can be tested and reused.

Service to validator

Services often depend on validators before they write state or emit output. That keeps bad input from reaching the runtime layer and makes the framework fail closed instead of guessing.

State to dashboard

Dashboards should read from workspace state rather than invent their own picture of the system. The dashboard relationship matters because it keeps the visible surface aligned with the real state underneath it.

Docs to registry

Docs pages should mirror the command registry or other canonical metadata instead of becoming a separate story. When docs and runtime disagree, the framework becomes harder to trust.

Plugin to core

Plugins should connect through defined contracts rather than reaching into the framework randomly. Good relationships keep plugins useful without letting them blur the ownership boundary.

Why this page matters

Understanding the relationships is what lets you predict the effect of a change before you make it. That is the difference between reactive maintenance and deliberate framework design.
