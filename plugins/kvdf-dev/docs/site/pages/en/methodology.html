KVDF Dev System

Methodology

KVDF development starts with a planning gate: define the change, check the boundary, slice the work, approve before execution, and verify after every slice. The method keeps the runtime, the docs, and the evidence aligned so framework change stays controlled instead of improvised.

Planning gate

The planning gate is the formal checkpoint that turns an idea into controlled framework work. It collects the scope, the boundary, the slice list, the approval state, and the next exact action so the framework can move only when the work is actually ready.

Plan first

Start by defining the change, the problem, and the result you want. Planning removes uncertainty before code changes begin and gives every later step a clear reason to exist. It also gives the reviewer a stable reference point instead of forcing them to infer the intent from code later.

Define scope

State exactly what is included and what is excluded before touching code. Clear scope prevents the request from silently growing beyond the approval that was granted, and it keeps the slice small enough to review honestly.

Boundary rules

Every framework change must stay inside the approved boundary unless the boundary is deliberately reopened. If a change crosses into another product, workspace, or surface scope, the work stops and returns to planning instead of widening silently.

Name the contract

List the commands, services, pages, schemas, or plugins the slice may touch. That keeps the work attached to concrete runtime surfaces instead of vague implementation talk, and it makes the review focused instead of speculative.

Slice the work

Break the change into small tasks that can be reviewed and finished one at a time. Slicing makes the work measurable, reviewable, and easier to stop if something drifts, especially when the change touches more than one framework layer.

Approve before execution

Nothing should move into implementation until the slice list is approved. Approval is the point where the plan becomes an execution path instead of a draft idea, and it is the moment when the team can hold the slice against the agreement.

One slice at a time

Implement only the current slice and stop before moving to the next one. Single-slice work preserves focus and prevents one task from silently swallowing another task?s scope, which keeps the change readable and traceable.

Verify

Confirm the slice with tests, checks, or acceptance criteria. Verification turns 'it looks done' into 'it is safe to continue' and protects the next slice from uncertain foundations. Without it, the work is only assumed, not proven.

Re-review on change

If scope changes, return to planning and approve the new slice list again. The old approval no longer describes the real work, so the change must re-enter the gate before it continues. Re-review keeps the approval trail honest when reality moves.

Stop rules

Stop immediately when the plan is incomplete, the boundary is unclear, or the slice is larger than the approved work. Stopping is the correct move when safety is missing, because continuing without a clean gate hides risk instead of resolving it.

Escalation rules

Escalate back to planning when the slice touches a new surface, changes scorecard meaning, or affects another framework boundary. Escalation prevents small local decisions from becoming hidden framework-wide changes and gives the owner a chance to re-evaluate the risk.

No hidden cleanup

Only work on the approved slice. Refactors and side fixes should wait for a later slice so the review trail stays honest and easy to follow. Hidden cleanup makes reviews blurry because it mixes the approved change with extra work nobody signed up for.

No memory-based changes

If the change is not in the approved plan, treat it as a new request. The system should obey the written plan, not the memory of the conversation or an assumption about what might be useful. This rule stops the project from becoming a loose chat-driven pile of opportunistic edits.
