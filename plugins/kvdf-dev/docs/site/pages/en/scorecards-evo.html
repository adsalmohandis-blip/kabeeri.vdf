Scorecards and Evolution

Scorecards begin as review signals for a module, workflow, or capability. When they are locked, they become traceable Evolution slices with clear scope and acceptance.

Scorecard Preparation Steps

1. Identify the module, flow, or capability being reviewed.

2. Name the source and target surfaces if the work crosses boundaries.

3. Describe what changed, or what is expected to change, in plain language.

4. Check the current evidence and the current state before judging the slice.

5. Generate the tasks, assess them, cover them with the right checks, and approve them.

6. Build the task packet so execution has a concrete unit to work from.

7. Verify the validator, policy gate, and docs that protect the relation.

8. Run batch execution on the approved work.

9. Read the workspace state and dashboard after execution.

10. Trace the result back to the module, flow, or capability that produced it.

11. If drift appears, return to planning or update the relationship pattern.

What scorecards are

Scorecards are the framework's review signal. They show readiness, risk, boundary fit, and confidence before a slice becomes implementation work. A scorecard is not a decorative note; it is the decision surface that tells the team whether the slice is actually ready to move.

What they measure

A useful scorecard measures clarity, scope control, evidence quality, readiness, risk, and how much trust the team should place in the current slice. It should tell the team not only the score itself, but why that score is safe or unsafe.

Lock before execution

Once reviewed and locked, a scorecard becomes a reliable improvement target. Locking means the evaluation can be trusted while work moves forward, and it prevents the score from drifting while the slice is underway.

Convert to Evolution

Locked scorecards can become tracked Evolution slices with explicit scope and acceptance. That is how review turns into a real implementation path instead of a loose note or a hidden backlog item.

Keep in sync

Scorecards, tasks, dashboard state, and Evolution state should agree. If those views disagree, the system is no longer describing the same reality, and the work should return to planning before it continues.

Why it matters

This is how review becomes tracked implementation instead of a vague note. The scorecard gives the framework a measurable way to decide what should change next and what should wait.

How it helps the framework

Scorecards identify where the framework is strong, where it is drifting, and what should be improved next. They make the framework honest about its own quality and keep Evolution focused on the right slice.

How it helps plugins

Plugin scorecards show whether a bundle is safe, consistent, reversible, and ready to extend the platform. That helps plugin development stay disciplined instead of ad hoc and keeps plugin work inside the right boundary.

How to use it

Read the scorecard, review the evidence, lock the slice when the assessment is stable, then turn the locked result into an Evolution task. That keeps review and implementation connected without mixing them.

What to avoid

Do not treat scorecards as decoration or as a second backlog. They are a decision surface, and they should only move forward when the evidence behind them is clear and the scope is still valid.
