Topic Details

Who this site is for

This site is for framework contributors, maintainers, and plugin authors who are changing Kabeeri itself. If you are working on the engine, the contract, or the developer method, this is the starting point.

What KVDF is

KVDF is the operating framework around Kabeeri. It defines how planning, approval, runtime state, validation, dashboards, plugins, and evolution stay in sync so the system remains strict instead of becoming a loose collection of tools.

How to read it

Start with Methodology, then read the four pillars, then move into framework development, plugin development, and scorecards. The site is arranged so each page builds on the previous one rather than repeating the same idea in a different form.

Methodology

The method is strict: plan first, define scope, slice the work, approve before execution, then stop after each slice and verify. That rule is the backbone of every framework change, because it keeps the runtime, the docs, and the evidence aligned.

What this site covers

The flow chains of the system, the main modules, module relationships, relationship patterns, framework development, plugin development, and the evolution loop all belong here. These are the pieces that explain the framework as a whole and how it changes safely over time.

What does not belong here

App-building guidance, workspace delivery, vibe developer instructions, and app scorecards belong in the vibe docs site. If the change is about a customer app rather than the framework itself, it should live elsewhere.

The working rule

If a change affects the framework contract, the runtime, or the developer workflow, it belongs in this site and must be reviewed as a framework change. That rule helps prevent accidental drift between what the framework promises and what it actually does.

Canonical links

Use the workflow instructions, governance readme, scorecard index, and implementation plan as the canonical references when you need the formal policy behind the site.
