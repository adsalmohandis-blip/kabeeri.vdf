What this report answers

What KVDF can do, which CLI surfaces map to those capabilities, how the plugin ecosystem is split, and how the Viber track gates app delivery before execution.

Primary operating split

Framework-owner work changes KVDF itself, Viber app-developer work uses KVDF to build apps, and the shared platform layer keeps both safe and resumable.

Important guardrail

Raw ideas do not go straight to execution. The Viber pipeline forces questionnaire, brief, docs, design, version, evolution, task punch, approval, and gate checks first.

Operating Model

Track Who it serves Core cycle Owns
Framework Owner Track Maintain and extend KVDF itself. resume -> evolution priorities -> evolution temp -> validate -> verify -> commit on main -> push origin main Evolution Steward, framework governance, release readiness, dependency mapping, and follow-up tasks for KVDF internals.
Viber App Developer Track Build real applications with KVDF as the engine. resume -> vibe/ask -> blueprint/questionnaire -> evolution -> temp -> tasks/capture -> handoff -> validate Vibe-first intake, milestone-based app evolution, app task execution, capture records, app blueprints, prompt packs, and local-first handoff.
Shared Platform Layer Both tracks. resume -> guard -> conflict scan -> validate -> dashboard/reports CLI engine, workspace state, task tracker, dashboard, sync preflight, security, tokens, locks, and runtime schemas.

Viber Track

Viber app delivery pipeline

The Viber pipeline is the gated path from raw app idea to execution. No answers means no real planning, no approved brief means no evolutions, and no task punch means no Codex execution.

Viber release train

The release train is the FIFO readiness model for app delivery. It moves through docs, evolution, task, validation, security, handoff, and publish readiness.

Stage Gate / step Meaning
1 idea Raw request enters the Viber pipeline, but cannot execute immediately.
2 questionnaire_generation The system asks for structured intake questions.
3 questionnaire_answers Answers are captured as durable planning input.
4 answer_completeness_check The answer set is checked for missing planning gaps.
5 brief_generation A brief is produced from the intake evidence.
6 brief_review The brief is reviewed before any further planning.
7 brief_approval Owner approval unlocks the downstream planning path.
8 state_resync Current local state is reconciled before planning continues.
9 current_state_report A current-state report becomes the planning anchor.
10 app_boundary The app boundary is defined so KVDF Core stays separate.
11 documentation_architecture The docs architecture is planned.
12 documentation_folders Foldered docs are materialized.
13 documentation_files The compatibility file list is derived.
14 system_design System design artifacts are prepared.
15 database_design Data model and persistence design are prepared.
16 ui_ux_design UX and UI design artifacts are prepared.
17 source_control_plan Delivery mode and source-control path are chosen.
18 security_plan Security posture and gates are planned.
19 version_plan Version strategy is proposed and reviewed.
20 evolutions Approved changes become tracked evolutions.
21 evolution_order_validation Evolution order is validated before tasking.
22 task_punches Task punches are generated only after approval.
23 task_punch_review The task punch set is reviewed.
24 approval The owner gate approves the next action.
25 materialization The approved work is materialized into workspace files.
26 codex_prompt The execution prompt is assembled.
27 security_gate Security checks block unsafe execution.
28 handoff_gate Handoff readiness is verified.
29 source_control_gate The mode-specific source-control gate is checked.
30 execution Codex execution begins only after the gates pass.
31 validation Validation checks confirm the result.
32 security_scan Security scanning runs after execution.
33 handoff The result is prepared for handoff.
34 dashboard_update Dashboards and live views are updated.
35 learning_capture Reusable learning is recorded.
36 closeout The session closes with a resumable final state.
