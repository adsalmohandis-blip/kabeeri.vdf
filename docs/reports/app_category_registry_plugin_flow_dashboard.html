A cleaner handoff from category planning to workspace building.

This dashboard visualizes the pipeline from app_category_registry to a selected domain plugin and then into app_folder_structure. The emphasis is on contract clarity, not just file generation.

Stage 1

Category planning

Resolve app type, questionnaire, specs, roadmap, and workspace plan.

Stage 2

Domain plugin

Keep blog, CRM, ecommerce, booking, and similar builders separate.

Stage 3

Workspace build

Materialize the governed app workspace only after the contract is valid.

Hard contract flow

The pipeline is intentionally fail-closed. If the handoff contract is missing or mismatched, app-folder creation stops instead of guessing.

How to read the chart

The registry decides the app shape, the domain plugin owns the category-specific behavior, and app_folder_structure turns the approved plan into a governed app workspace.

  • app_category_registry writes the plan and contract.
  • The selected domain plugin fills in domain-specific behavior.
  • app_folder_structure only materializes once the contract validates.
  • Promotion is separate and remains owner-controlled.

The dashboard visuals now follow the plugin’s own output: dashboard product shape, enterprise clean style, technical palette, accessible data surfaces, and a bento-grid presentation pattern.

If Mermaid fails to load, the flow is still the same: app_category_registry → Selected domain plugin → app_folder_structure. The approved promotion target remains ./plugins/<plugin-slug>/, while the workspace stays private until promotion.
